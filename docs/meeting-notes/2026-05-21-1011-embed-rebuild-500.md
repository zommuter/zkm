### Agenda 3 — Resilience zkm carries regardless of root cause

**Code facts (embed.py):** retry loop 264–288: `for attempt in range(3): if attempt: time.sleep(5*attempt)` = 0/5/10s = 15s total, then `raise EmbedUnavailable` aborts. Checkpoint at `done % checkpoint_every < _EMBED_BATCH` (297), checkpoint_every=500. `_EMBED_BATCH=32`. Server stays 500 >15s AND wall at fixed absolute ~864 → resume re-hits same wall; no run gets past it via cross-run resume.

🔧 **Quinn:** Actual unblock = let the SAME run wait out the ~30–40s self-recovery (attempts 0/15/30/60s) so it continues past 864 in-process.

🏗️ **Archie:** Two cheap permanent fixes: (1) widen in-process backoff to span recovery; (2) lower checkpoint_every to bound re-work.

😈 **Riku:** Long backoff adds wall-clock (~60 stalls × ~1min ≈ +1h one-time rebuild) — tolerable, ugly steady-state. MASKS the server bug → keep the zomni root-cause item open; "backoff works" must NOT close it.

✂️ **Petra:** Tight scope: (1) widen escalating backoff; (2) lower checkpoint_every; (3) log each stall (position + wait + body). OUT: adaptive batch sizing, circuit-breaker, zkm-side model restart. `_EMBED_BATCH` reduction stays a diagnostic experiment, not permanent.

🔧 **Quinn:** Widened backoff IS the recovery-window test (next-step #3); instrument each stall so the zomni item gets stall-frequency evidence.

**DECISION (Zommuter):** Backoff + checkpoint + log. Widen escalating backoff (~40s span), lower checkpoint_every, log each stall (position + wait + body). `_EMBED_BATCH` stays diagnostic-only. Stall log doubles as the D1 diagnostic — one `zkm index` run diagnoses and attempts the unblock simultaneously.
