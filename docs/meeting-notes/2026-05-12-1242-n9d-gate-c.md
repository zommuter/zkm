### Item 2 — Are we attacking the right problem? (state-of-the-art audit)

**Archie:** First, the methodology caveat is real and worth recording. The classifier UI shows ~120 chars of `context_snippet` for each item — for very short ambiguous strings (2-char organisation acronyms vs reference-number prefixes), that's often not enough to tell apart from context. Some fraction of the 12 FPs may be misclassified by the human gate itself. So FP-drop=20% is an *upper bound* on the verifier's error rate; the true rate could be lower. But not enough to get under 5%.

**Mira:** State-of-the-art for *personal-corpus* entity extraction (not academic NER on CoNLL news) splits into three established recipes:
1. **Structured-first extraction.** Apple Mail's Data Detectors, Gmail's "smart" extraction (flight numbers, packages, calendar invites), Microsoft Graph — all of these lean on *known structure*: email headers (From/To/Cc), signature blocks, known-template senders, iCalendar attachments. Body NER is a *supplement*, not the source.
2. **Pretrained NER + rule overlay.** spaCy / Stanford CoreNLP style: take a pretrained model, layer regex patterns for closed-set classes (email, phone, URL, dates), apply a stoplist. F1 in the 75–85% range on email-style text. This is what zkm currently does.
3. **Active learning + custom rules.** Production systems iterate: surface FP/FN cases via human review, retrain or extend rules. The `scrub` + verifier loop is conceptually this — but as a *batch retrofit* rather than an online loop.

The verifier-as-LLM-postfilter approach (what N9d is) is *less common* in production — it appears in research papers but rarely in shipping products. Production prefers rules + retraining over LLM gating because LLM verifiers have exactly the failure mode we just observed: small-model context-insufficiency on short ambiguous strings.

**Mira (continued):** Where zkm stands:
- **Adapting established results sensibly** for the core extraction pipeline: spaCy + pattern overlay + stoplist matches recipe 2 above. This is well-trodden ground.
- **Missing recipe 1 partially.** Structured extraction from email headers is in scope but the value lever needs to be located precisely (see Archie's correction below).
- **N9d (LLM verifier) is the unusual choice** — closer to research than to production. The Gate C result confirms the production wisdom: small-LLM verifiers don't reliably outperform stoplist+POS on the cases where stoplist+POS already struggles.

**Riku:** This reframes the question. The pilot's failure isn't "we picked a bad verifier model" — it's "we tried to use a verifier to clean up a problem that the extraction pipeline shouldn't have created in the first place." A structured pre-extractor would *prevent* the FPs rather than *post-filter* them. That's a categorical improvement, not a tuning improvement.

### Item 3 — Onboarding new experts; action-item plan

**Nora (new — IE / NER typology lens):** Quick framing for the entity-vs-data question: classical IE distinguishes three kinds of extraction targets — (1) *named entities* (things with potential identity in a KB: persons, organisations, places); (2) *temporal/numeric expressions* (amounts, dates, durations — values, not identities); (3) *relations / slots* (e.g. `Person works_at Organization`). Schema.org and Wikidata follow the same split: a `Person` has an ID (`Q42`); a currency amount is a *literal value* attached to a transaction, not a `Q…` ID. The pilot's noise classes map onto this cleanly — currency amounts and reference-number patterns are *values*, not entities; geographic abbreviations are place entities. The verifier was being asked to make the entity/value distinction on a 2-char string with 120 chars of context. That's the *hardest* version of the task, where even humans require domain knowledge.

**Pim (new — PIM engineering lens):** On signatures and structured-first: every major email client I know about does two things zkm-eml doesn't yet. (1) **Signature detection** — heuristics: `^--\s*$` separator, `Mit freundlichen Grüßen` / `Best regards` / `Sent from my` / `Gesendet von` markers, then everything below up to ~10 lines is the signature block. Extract `Person` + `Phone` + `Email` + `URL` from there at near-100% precision because the *position* is the signal. (2) **Quoted-reply stripping** — lines starting `>` or after `On <date> X wrote:` are previous-thread content, not the current author's. Re-NER on quoted text is double-counting plus extra noise. Both are pre-NER text-cleanup, not entity-resolution — cheap to add, no model retraining.

## Decisions

- **N9d-as-designed is closed.** Gate C fired (FP-drop=20%, well above the 5% threshold). The upfront gate is honoured. Closure is not a quiet pivot — the *question* of whether to clean up these FPs at all is reopened, not silently re-routed to a successor project.
- **The verifier failure is category-correlated, not uniform.** Per `suspicious_reason`: `1-char` drops and `single-token MISC` drops achieved 0% FP; `person-lowercase` (the original N9d motivating class) showed 50% FP; `2-char` showed 31% FP. The verifier fails worst exactly where it was supposed to help most.
- **Both ends of the confusion matrix are broken.** FN-keep=37.5% means the verifier also misses junk in its KEEP verdicts (`single-token MISC` 100% FN; `no alphabetic content` 56% FN). The verifier-postfilter design has bidirectional failure, not just precision miscalibration.
- **The LLM-verifier-as-postfilter design is research, not production state-of-the-art** for personal-corpus extraction. Production recipes lean on structured-first extraction (recipe 1) and rule overlays (recipe 2); the active-learning loop (recipe 3) handles residuals. Verifier-postfilter on small models has the failure mode N9d hit.
- **The entity-vs-data scope question is load-bearing on any future verifier or scrub design.** Some pilot FPs are FPs only because we have not decided whether reference numbers and currency amounts are entities. Resolve scope before designing more cleanup mechanisms.
- **Signature-block extraction (N9g-pre) is unblocked.** Position-based extraction is high-precision regardless of how entity-vs-data resolves, so it does not depend on the upstream meeting.
- **General body-NER cleanup (N9g) is blocked on the entity-vs-data meeting.** Designing it before scope is set risks re-creating the FPs we'd need to clean up.
- **N9e gate condition is moot.** N9e was gated on "(N9d shipped) AND (≥5 verifier-override cases in Stage 2 pilot)." N9d did not ship; the gate cannot fire. N9e remains in backlog but with no live trigger path.
- **Future pilot gates must include ≥1 paragraph of context per item** (methodology rider). The 120-char `context_snippet` in this pilot was insufficient for confident human judgement on short ambiguous strings — propagating that limitation to future gates would re-create the same uncertainty.
- **Out of scope for this meeting:** model swap retest (deferred unless evidence reopens N9d), typed-slot data model (defer to entity-vs-data meeting), `zkm-eml` quoted-reply stripping (related but distinct from signatures; capture as separate TODO if confirmed valuable in N9g-pre design).
