## Discussion

### Item 1 — Frame the problem

**Archie:** The amendment contract gives `entities` set-union semantics with "existing entries take precedence to keep order stable" (`src/zkm/amendments.py:121` `merge_fields`). Correct for enrichment. But N9b just demonstrated the failure case: when an amender's extraction *quality improves* and the new (clean) output is a strict subset of the old (dirty) output, set-union preserves the dirty surplus forever. The textfilter is correctly wired, the cache correctly invalidated (`+textfilter-v1` in `version.py`), the new extraction is verified clean — but 16,138 "Subject"/loc entries from the pre-textfilter run survive in frontmatter because `apply_queue` cannot remove what extract no longer emits.

**Riku:** Not a bug — a deliberate property. Set-union is what makes amenders commutative across plugins. If we add "replace" semantics, silence from one amender becomes ambiguous (does it mean "delete everything I ever added"?). Contract gets messier fast.

**Petra:** Minimum to unblock pilot (closes 2026-06-05): a scrub script that walks `*.md`, drops stoplist matches, writes back atomically. Replace-mode is a real design conversation that takes a session and touches the contract.

**Archie:** Three ownership options: (a) `plugins/zkm-ner/scripts/scrub_stoplist.py`; (b) `src/zkm/scrub.py` + `zkm scrub` CLI; (c) "negative amendment" records — replace-mode through the back door.

**Riku:** Option (c) still needs a one-time pass to generate the negative records. Saves nothing; adds contract burden.

### Item 2 — Scope comparison

**Petra:** Option 1 (plugin script): ~25 LOC + 3 tests, ~0.5s. Option 2 (replace-mode): ~80 LOC + 8 tests + spec update, ~1.5s. Replace-mode would let zkm-ner own `entities` exclusively ("single producer per field") — cleaner contract but a migration.

**Riku:** "Single producer per field" is a real design statement worth making explicitly — but not without evidence we need it.

### Item 3 — N=2 check

**Petra:** N9c (POS-filter for "Du", "wünschen", "EUR") will hit the same retroactive-cleanup problem. If scrub is the answer for N9b, it will be again for N9c.

**Archie:** N=2 hit: N9b stoplist + N9c POS-filter both want to remove previously-emitted entities. A generic scrub utility taking a plugin-supplied predicate is N=2-justified.

**Petra:** Counter: if N9c is applied during *extraction*, the new extraction won't emit "Du" and a replace-mode re-run handles cleanup automatically. The N=2 is real only if we lock in scrub-as-mechanism.

**Archie:** Petra's point flips the calculus. Option 2 makes cleanup implicit (re-run = clean). Option 1 makes it explicit (run scrub when needed). For a personal corpus with manual verification, explicit is more debuggable.

**Riku:** Observe-before-preventing: if we scrub now, the next amender that wants "owns this field" re-surfaces the issue and we meet then. Replace-mode now is designing for a second amender we don't have yet.

**User:** Scrub now, defer replace-mode.

## Decisions

1. **Scrub mechanism = `zkm scrub <plugin>` core CLI.** N=2 justified by N9b stoplist + future N9c POS-filter.
2. **Plugin contract addition:** optional `scrub(store_path, config, *, dry_run, progress=None) -> dict[str, int]`. Plugins without `scrub` → exit 2.
3. **CLI shape:** `zkm scrub <plugin> [--dry-run|--apply] [--verbose]`. Default `--dry-run`. No auto-commit. Mirrors `zkm rm`/`zkm gc`.
4. **Sidecar boundary:** `scrub()` MUST NOT mutate `<md>.amendments.json`. Documented in `docs/plugin-spec.md`.
5. **Idempotency requirement:** plugin's `scrub()` is idempotent; second `--apply` run reports 0 changes. Tested.
6. **zkm-ner `scrub()` v1:** drops entities where `e["value"].strip().lower() in _STOPLIST`. Type-agnostic, case-insensitive.
7. **Versioning:** core `0.2.0 → 0.3.0`; plugin requires-clauses cascade `<0.3.0 → <0.4.0` across all 6 plugins. zkm-ner bumps to `0.2.0` (new scrub feature = minor); other 5 plugins patch-bump for requires-clause only. All bumps tagged per bump-and-tag rule.
8. **Commit sequencing:** feature commit (scrub.py + cli + tests + docs + v0.3.0 tag) first; cascade commit (6 plugin pyprojects + v tags) follows.
9. **Pilot deadline:** scrub run + re-pilot before 2026-06-05; mark N9b-5 done after delta verified.

**Explicitly out of scope:**
- Amendment replace-mode redesign (deferred to backlog meeting).
- Negative-amendment record shape.
- Generic `--predicate` CLI flag (plugin owns its predicate).
- Auto-commit on `--apply`.
